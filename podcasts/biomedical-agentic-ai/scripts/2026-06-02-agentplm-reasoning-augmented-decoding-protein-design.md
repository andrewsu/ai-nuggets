# AgentPLM — reasoning-augmented decoding turns protein language models into agentic designers

Paper link: https://arxiv.org/abs/2606.02386

## Script

Today's nugget is AgentPLM, posted to arXiv yesterday, which proposes a clean and aggressive idea: protein language models should stop being passive oracles and start behaving like agents that can interrogate biophysical tools mid-generation. The reason this is worth a few minutes of attention is less about the headline benchmark numbers — which are interesting but live in the usual cluster of antibody and enzyme tasks — and more about the architectural pattern, which is broadly transferable to other agentic AI systems in biomedicine.

Here is the setup. A protein language model, the way it is normally used, is a one-shot oracle. You hand it a prompt or a partial sequence, it runs a single forward pass, and out comes a candidate sequence. If that candidate violates a thermodynamic constraint, or fails to dock the way you wanted, or folds into the wrong shape, the model has no idea — and even if you score it externally with something like a folding predictor or a docking engine, that signal does not loop back into how the next sequence is generated. You end up in the usual emit-then-filter pattern, where the model proposes blindly and an outer evaluator quietly throws away most of what it produces. That works, but it wastes the model's own ability to course-correct.

AgentPLM does something different. It introduces what the authors call Reasoning-Augmented Decoding — R-A-D for short — which interleaves the model's autoregressive generation with tool calls to three standard biophysical oracles: E-S-M-Fold for structure prediction, Fold-X for thermodynamic stability, and Auto-Dock Vina for docking. So instead of generating a full sequence and only then checking it, the model generates a span, calls a tool, reads the result back into its own context, and then decides whether to continue, revise, or escalate. The decoder is, in effect, doing online error correction without any explicit backtracking machinery — the tool calls themselves are the correction signal.

The second piece is the training story, which is where this work goes beyond just bolting tools onto an existing protein language model. They introduce Contrastive Agent Policy Optimisation, C-A-P-O, which is a trajectory-level extension of direct preference optimisation. The thing C-A-P-O is trained to learn — and this is the subtle point — is not just which sequences are good. It is when oracle feedback is actually informative. That distinction matters a lot. If you train a model to always call the folding predictor, you get an expensive system that makes lots of redundant calls on easy cases. If you train it to never call, you are back to the passive oracle. C-A-P-O is supposed to teach the policy to call tools selectively, where the marginal information from a tool exceeds its cost. That is the right framing, and it is a recurring pattern across agentic systems — the hard problem is not tool use, it is metacognitive: knowing when a tool call is worth the round trip.

The evaluation covers the standard battery — de novo enzyme design, antibody optimisation, thermostability, protein-protein interaction interface design, and zero-shot fitness prediction — using standardised oracle APIs and controlled sequence-identity splits so the gains do not come from memorisation. The headline result the authors highlight is state of the art on antibody top-ten-percent hit rate against the strongest passive baseline. I want to be honest that the abstract gives the qualitative claim more than a precise number, and that is a place where a careful reader should dig into the actual tables before getting too excited.

A few read-throughs that matter for this audience. First, the agent-inside-the-decoder pattern is the part to pay attention to, more than the specific protein tasks. The same shape transfers to any domain where you have a generator and cheap-enough evaluators — molecule design with quantum-chemistry or docking oracles, materials with simulation oracles, even pipeline construction with execution feedback. Anywhere you currently run an outer scoring loop that throws away most candidates, there is a candidate refactor where the evaluator becomes an in-decoder tool and the model learns when to ask it. Second, the C-A-P-O framing — train the policy to predict when feedback is informative, not just imitate high-fitness outputs — is the more durable contribution than R-A-D itself. R-A-D is a plumbing choice; the question of when to invoke an expensive oracle is a question that every agentic system eventually has to answer, and a trajectory-level preference learning method is a reasonable formalisation of it.

One caveat worth naming. The author list is small and the institutional context is not visible from the abstract, so the depth of the empirical validation is worth checking before declaring this a milestone. The pattern is interesting whether or not the benchmark margins survive scrutiny — but a reader who wants to build on this should look hard at the ablation on tool-call frequency, because that is the experiment that distinguishes "the model genuinely learned when to ask" from "the model just got a free boost by calling tools on every step."

That is today's nugget.
