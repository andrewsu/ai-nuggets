# Clair3-Connect — re-architecting a long-read variant caller as a developer-built agent interface: 60-for-60 correct, 7-to-14x fewer tokens than the shell wrapper

Paper link: https://www.biorxiv.org/content/10.64898/2026.06.25.734665v1

## Script

Today's pick is a bioRxiv preprint posted yesterday from Ruibang Luo's group at the University of Hong Kong, and it's a paper that argues something I think is correct and that the agentic bioinformatics community has been mostly hand-waving past. The title is "Client-server interfaces enable efficient agent-driven variant calling," and the central claim is that when an LLM agent drives a bioinformatics tool, the way the tool exposes itself to the agent matters as much as the algorithm inside. The corollary is sharper. Agentic interfaces should be a first-class deliverable of bioinformatics tool development, written by the developers, not glued on after the fact by a third party.

The setup. There is now a wave of agentic systems that drive existing bioinformatics tools. The agents read the README, figure out the install, parse the help text, infer the right flags, run the tool in a shell, and pick up the results. Each of those steps costs turns, tokens, and tool calls, and each of them is a hallucination surface. The agent is reasoning about a tool that was built for a human expert who already knows what is conventional. The defaults, the file conventions, the quirks the developers know not to violate — those are not in the documentation, they are in the developers' heads. A shell-level agent rediscovers them, sometimes wrong, every run.

What they did. They took Clair3, a widely used deep-learning-based long-read variant caller, and re-architected it as a client-server system called Clair3-Connect. The client does all the genomics-side processing — pileup, candidate generation, post-processing — and holds the identifiable data. The server runs only the neural network inference. The client sends feature tensors to the server. Sample identifiers and genomic context never leave the client. On top of that, the client exposes a small set of agent-facing tools with strict JSON schemas. The agent does not write a shell command. It picks a tool, fills in the schema, gets back a structured result, and moves on.

The numbers are clean. On an APOE diplotyping task, all sixty agent runs were correct. The agentic interface used about twelve thousand tokens across three turns. The shell-driven baselines — the same task, same model, the agent doing it the old way — used between eighty-one thousand and one hundred sixty-three thousand tokens. That's six-point-eight to fourteen times fewer tokens, at about a quarter of the wall-clock time, with token usage that varied by only four percent across runs instead of thirty-five percent. The interface didn't just save cost; it made the agent stable.

The architectural concessions are worth naming because they are honest. Keeping the client light meant dropping the local pileup and phasing stages from the inference path; SNP F1 stayed within zero point one to zero point three points of standard Clair3 by fifty-times coverage, which is acceptable but not free. Adding mutual TLS and AES-256-GCM encryption to the client-server channel cost about seven percent of end-to-end runtime — a real-world price for the data-residency split, which is the right design for a system that touches identifiable human genomic data. The server never sees the sample ID or the genomic context, only the feature tensors it has to score; that boundary buys a meaningful privacy posture without sacrificing the model.

Now, the editorial reads, because this one matters for how the broader field is going to evolve.

First. The reason a developer-built agent interface dominates a third-party shell wrapper is not that the wrapper is poorly written. It's that the wrapper cannot recover defaults and conventions only the developers know. The Clair3 team knows which arguments are practically required versus nominally optional. They know the pileup stage's coverage thresholds were tuned for a particular use case. They know the file conventions the help text doesn't bother to spell out. When the developer builds the agent interface, those decisions get baked in as the structured tool's defaults. When a third-party wrapper builds it, those decisions are rediscovered, badly, in the agent's reasoning loop. That asymmetry is the case for asking bioinformatics tool authors to ship an agent interface alongside the CLI, not as an afterthought.

Second. The client-server split is doing two jobs at once and it's worth seeing both. It is a structured interface for the agent, which is the efficiency story. It is also a data-residency boundary, which is the privacy story. Sample identifiers and genomic context stay client-side; only inference tensors cross the boundary. That means a centralized inference server — a shared GPU somewhere — can serve many independent client institutions without ever seeing identifiable patient data. For long-read variant calling specifically, which is increasingly used in rare disease and clinical genomics, that is a serious unlock. The seven percent encryption tax is a rounding error against the alternative of every institution standing up its own GPU stack.

Third. The lesson generalizes well beyond Clair3. The pattern is recognizable — schema-defined tool surface, data-aware client-side preprocessing, narrow server-side inference, deterministic structured outputs the agent can route on without parsing prose. Apply that pattern to alignment, to single-cell quality control, to motif discovery, to any bioinformatics tool that an agent is going to drive in production. The same efficiency, the same stability, the same provenance. The corollary I would draw is that the agentic-tool ecosystem will bifurcate. The tools whose maintainers ship native agent interfaces will become the agent-default; the tools that stay CLI-only will get the third-party wrapper treatment and pay the seven-to-fourteen-times token tax. That bifurcation is going to shape which bioinformatics tools the next generation of agentic systems actually call.

Fourth, the methodological caveat. The headline result is one task on one tool with one model. The authors are clear about this — APOE diplotyping is the demonstration, not the breadth. The architectural argument is more general than the benchmark, and the benchmark exists to make the architectural argument concrete. That is the right read.

The take. This is a paper that should land on the desk of anyone publishing or maintaining a bioinformatics tool. The agent-driven analysis pipeline is not a future state — it is the current state, today, and the design choice the field has been quietly punting on is how those tools should be exposed to agents. The answer this paper gives is the right one. Build the interface yourself, expose it as schema-defined tools, split the data residency boundary at the same time, and treat the agentic interface as a first-class deliverable of the tool. The cost saving is real. The reliability gain is the bigger story.
