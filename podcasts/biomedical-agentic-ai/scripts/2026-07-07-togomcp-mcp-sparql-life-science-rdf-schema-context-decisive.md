# TogoMCP — MCP-driven natural-language SPARQL over 23 life-science RDF databases, and why schema files are the only load-bearing component

Paper link: https://www.biorxiv.org/content/10.64898/2026.03.19.713030v1

## Script

Today's pick is TogoMCP, a new bioRxiv preprint from the Database Center for Life Science in Japan, with collaborators in Spain, that turns Anthropic's Model Context Protocol into a practical natural-language interface over life-science RDF knowledge graphs. There is a lot of MCP hype right now, but what this paper actually delivers is a clean, ablation-driven answer to a question the field has been muttering about for a year: when you wire a language model to a large structured knowledge base, what is the piece that actually makes it work?

The setup. The DBCLS RDF Portal aggregates more than seventy life-science databases — UniProt with its two hundred and seventeen billion triples, ChEMBL, MeSH, Gene Ontology, MONDO, Reactome, and so on — into a single SPARQL-queryable knowledge graph. In principle you can ask, in one query, for the FDA-approved drugs targeting human cancer-pathway proteins with known three-dimensional structures. In practice, you need to write SPARQL and know the per-database RDF vocabulary of every source you touch, and that gate keeps the whole resource out of reach for most biologists. Off-the-shelf language models are the obvious escape hatch, and they fail in a specific and instructive way: they generate syntactically valid SPARQL that fabricates predicates and misuses vocabulary, because the schema of every RDF endpoint was never in their pre-training data.

TogoMCP's response is a two-stage architecture running Claude — Sonnet four-point-five for the runs, judged by Opus four-point-six — as a protocol-driven inference engine. Stage one, entity resolution: the agent hits specialised REST APIs — UniProt search, ChEMBL molecule and target search, PubChem, PubMed, MeSH descriptors, NCBI E-utilities, and TogoID for cross-database ID conversion — to turn a natural-language entity like aspirin into the specific identifier the target database expects, PubChem CID two-two-four-four in that example. Stage two, schema-guided SPARQL: for each database it is about to query, the agent pulls what the authors call a Metadata-Interoperability-Exchange file — MIE for short — a hand-curated four-to-six-hundred-line YAML per database containing ShEx shape expressions of the actual predicates and IRIs, sample triples, seven tested example SPARQL queries at three complexity levels, cross-database join patterns, and documented anti-patterns. That MIE file gets dropped into the model's context on demand, and the SPARQL is generated against it.

The benchmark. Fifty biologically grounded questions across five types — yes-no, factoid, list, summary, and choice — with every question required to actually need database access to answer, verified against PubMed to rule out anything recoverable from training data. Sixty percent of questions span two or more databases; twenty percent span three or more; every one of the twenty-three currently supported databases is exercised at least once. Each answer is scored five times by an LLM judge along four axes — recall, precision, non-redundancy, readability, five points each — for a two-hundred-fifty question-run corpus per condition.

The headline number is that the TogoMCP agent beats an unaided Claude baseline by two-point-seven-two points on a twenty-point scale, Cohen's d of point-nine-two, statistically hard. Win rate seventy-five percent, tie ten, loss fourteen. But the number that matters is in the ablation, and it is the reason to spend five minutes on this paper.

The authors built out an elaborate behavioural protocol they call the Usage Guide: question classification into four categories, per-type SPARQL budgets, tool prioritisation rankings, defensive querying with limit-ten exploratory calls, output discipline rules, mandatory workflow steps. All the scaffolding a competent prompt engineer would build. And it turns out that ripping out the entire Usage Guide and replacing it with a single system-prompt sentence — call list-databases and get-MIE-file before you query — recovers the full benefit. Delta plus two-point-seven-three versus plus two-point-seven-two. No measurable difference. The elaborate protocol contributes zero above the one-line instruction.

Now take the schema file away entirely. Improvement collapses from plus two-point-seven-two to plus point-three-zero, statistical noise. The model, deprived of the schema context, roughly doubles its tool-call count trying to compensate through trial-and-error SPARQL, and on list and summary questions actually underperforms the no-tool baseline. Sixty-two percent of runs in that condition burn eighteen or more tool calls and get worse answers for it. Tool-call count is negatively correlated with score, correlation coefficient minus point-three-seven-five — runaway agent loops are a signal of struggle, not thoroughness.

The general principle the authors extract, and I think it earns the framing, is this: for tool-augmented LLM systems over structured knowledge bases, concise, dynamically delivered schema context is more valuable than complex orchestration logic. Give the model a good vocabulary reference at the moment of query and it does the work. Give it an elaborate playbook without the vocabulary and it thrashes.

Two supporting findings deserve mention. First, the two-stage ordering — entity resolution APIs first, SPARQL second — beat the reverse by one-point-six points, and on a complex cross-database query the API-first approach found seventy-five UniProt entries where SPARQL string filtering found three. Making the model do entity matching inside SPARQL text filters is the wrong regime. Second, structured lookup tools that return precise identifiers — PubChem compound IDs, MeSH descriptors — outscored free-text search tools by three or four points; when the retrieval is noisy, the downstream SPARQL is noisy too.

Caveats. Fifty questions, one LLM family, LLM-judge scoring rather than human expert review. Summary questions only budge half a point and still lose about a third of the time. Ninety-six seconds and forty-three cents per query, eleven and eighty-one times the baseline respectively; the authors argue that is justified for precision-critical queries the model cannot answer alone, and I would agree with the added caveat that any deployed version needs an early-exit when the tool-call count starts running away.

What to take away if you are building over your own knowledge graph. Spend the effort on curated per-database schema documentation — shape expressions, half a dozen tested example queries, a short list of anti-patterns — delivered on demand into the model's context. Do not spend it on baroque behavioural prompts. The model already reasons fine; it just needs the vocabulary.

That's TogoMCP.
