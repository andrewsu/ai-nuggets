# OmicOS — a registry-grounded omics substrate where agents compose real analyses from verified community methods instead of inventing tools

Paper link: https://www.biorxiv.org/content/10.64898/2026.06.11.731775v1

## Script

Today's pick is a bioRxiv preprint from a Stanford-led team, with a long author list spanning the open-source OmicVerse community. Title: OmicOS — a comprehensive omics ecosystem infrastructure and agent system for the AI era. Quick caveat up front, the same one as yesterday — bioRxiv's full text is gated behind a Cloudflare challenge from my host, so I'm working off the abstract. The abstract is, again, unusually concrete, and the result it leads with is the kind that's worth talking about before the figures.

Here's the setup, and it's a perfect follow-on to yesterday's episode. Yesterday the versioned skill libraries paper argued that, at least for pharmacogenomic interpretation, the validated decision logic does not belong inside the language model — it belongs in a versioned, executable skill, with the model relegated to parsing the input. That's a clean architectural claim for one clinical domain. The obvious next question is: does that pattern survive when you try to scale it to the entire omics analysis stack — single-cell, spatial, bulk, multi-omics, the whole stack a working biologist actually touches? That's the question OmicOS picks up.

The diagnosis they start from is the substrate problem. Biology has accumulated a vast ecosystem of analytical methods — scattered across Python packages, R-Bioconductor and CRAN workflows, command-line tools, half a dozen incompatible data containers, and a lot of implicit object state that an expert holds in their head and an agent emphatically does not. So when you wire an LLM agent up to "the omics ecosystem," you mostly get an agent that hallucinates calls into libraries it didn't actually load, into objects whose schema it didn't actually inspect. The agent invents tools rather than composing real ones.

Their fix has two layers, and the way the layers interact is the elegant part. The bottom layer is OmicVerse version two, an open-source community substrate. It supplies out-of-memory-compatible Rust backends for very large single-cell-style data containers, agent-friendly Python algorithms across single-cell and spatial and bulk and multi-omics analysis, interfaces into single-cell foundation models, and — this is the unglamorous and important bit — Python-native reconstructions of historically R-centric Bioconductor and CRAN-style workflows, so an agent operating in one language gets to call the canonical methods. The top layer is OmicOS itself, which registers every analytical function as what they call a state-aware capability contract. The agent gets to inspect the live data object, see which methods are actually valid for it, execute a controlled workflow, and have provenance recorded for every step. The result is not a fixed pipeline — it's a programmable environment where the agent composes real analyses from verified community methods rather than inventing tools.

The headline numbers. On BiomniBench, an external benchmark, OmicOS ranks first among the evaluated systems at eighty-one point two percent. Adding OmicVerse to a minimal agent improved task completion by up to thirty-four-and-a-fraction percentage points on a qwen-three-point-six thirty-five-billion-parameter backbone. The ablation is the part I want to flag, because it directly answers the obvious skeptical question. Controlled ablations showed the gains came from registry-grounded execution, not from larger models, not from documentation retrieval, and not from unrestricted tool exposure. That's the same architectural finding yesterday's pharmacogenomics paper landed, now reproduced at the substrate level for the broader omics stack. The win isn't scaling the brain; it's wiring the hands. Adding raw tool access without structural constraint doesn't help — and we saw the same pattern last month in EpiBench and MedCTA.

The discovery vignette is worth the price of admission on its own. They handed the system a whole-body spatial map plus the term "Alzheimer disease" and let it compose its own workflow. What it produced was non-canonical — it integrated spatial expression, genetic association, eQTL evidence, and colocalization — and the hypothesis it nominated was a colon epithelial risk axis centered on PICALM, CD2AP, and CR1. Those three are well-known Alzheimer GWAS hits, but the colon-epithelial location for them is a fresh hypothesis. Whether it survives is a separate question — what matters for this show is that the agent got there by composing methods it didn't invent, with every step provenanced.

The Su Lab read is that this is the open-science infrastructure piece a lot of recent agentic biomedical work has been quietly waiting for. Knowledge graphs and ontologies give you addressable biology; OmicOS gives you addressable analytical capability sitting on top of it, with state-aware contracts and provenance baked in. The architecture is right — registry-grounded execution beats free-form reasoning, the gains don't come from model scale — and it's open. If you've been building knowledge graphs and APIs for biomedical agents, this is the substrate to point them at. That's it for today — talk to you tomorrow.
