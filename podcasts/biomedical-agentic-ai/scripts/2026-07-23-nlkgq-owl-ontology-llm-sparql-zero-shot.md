# NLKGQ — a well-designed OWL ontology lets local LLMs write correct SPARQL zero-shot, no agents or RAG needed

Paper link: https://arxiv.org/abs/2607.18029

## Script

Today's pick is a small, contrarian paper out of the Max Planck Institute for Biological Cybernetics in Tübingen, and I want to feature it precisely because it argues against the reflex this whole show tends to celebrate. The reflex is: point a language model at a database, and if it can't answer, wrap the database in agents — a planner, a retriever, a query-writer, a critic — and orchestrate. This paper says, before you reach for any of that, fix your ontology, and the agents may turn out to be unnecessary.

The system is called NLKGQ, for Natural Language Knowledge Graph Query. The problem it solves is one every data-rich lab has. You've got a large research archive, and researchers want to ask ad-hoc questions about its contents — how many subjects, which acquisition parameters, what overlaps across studies — but writing the structured query takes expertise most of them don't have. The demonstration domain is neuroimaging: a hundred-and-eighty-terabyte MRI archive, eighty active studies, two thousand experiments, all described by a knowledge graph of about ten million triples. A researcher types a question in plain English, a language model translates it to SPARQL, the query runs against the graph, and the answer comes back.

Here's the claim, and it's a strong one. With a well-designed OWL ontology — that's the Web Ontology Language — behind the graph, an ordinary local language model generates correct SPARQL zero-shot. No fine-tuning, no retrieval augmentation, no multi-agent orchestration. On a set of twenty-one competency and regression questions built with domain experts, the best configuration got a hundred percent. Not "impressive for zero-shot" — a hundred percent, every question.

But the accuracy number isn't the interesting part. The interesting part is the ablation, because it tells you where the intelligence actually lives. They ran the same task across eight different representations of the same ontology, holding the graph and the questions fixed. Full OWL, with human-readable entity names and natural-language annotations — the comment and label fields that describe what each class and property mean — scored a hundred percent. Strip out just those annotations and it drops to eighty-one. Compress the names into terse codes and it falls further. Reduce the ontology to abstract graph structure with opaque identifiers, and accuracy collapses into the teens.

Sit with that. The dominant factor in whether the model writes correct queries was not the model, and not the prompt engineering — it was whether the ontology used full English words and carried its meaning in human-readable annotations. A property called "has acquisition time," spelled out, not a cryptic abbreviation. An explicit domain and range on every property. Meaningful names, no opaque codes. The model isn't reasoning its way past a badly-named schema; it's reading the schema the way a new lab member would, and a schema written for humans is a schema a language model can query. The semantics you painstakingly encode in an ontology are exactly the context the model needs — you just have to leave them legible.

There's a second finding I think is quietly important for anyone choosing a backend. They compared SPARQL over the OWL graph against auto-generated SQL over the same data in a relational database. Same questions, same models. SPARQL hit a hundred percent; the best SQL configuration managed fifty-seven. The reason is structural. OWL's explicit domain and range declarations tell the model how entities connect, so it gets join direction and relationship traversal right. A SQL schema — a pile of tables and foreign keys with no declared semantics — leaves the model guessing which way a relationship points, and it guesses wrong. If you're building natural-language query over your data and you have a choice of backend, that's a real argument for a semantic graph over a relational dump.

A couple of things ground this in the real world rather than a benchmark. The models are all local, open-weight Qwen variants running on modest institutional hardware — a deliberate choice, because this is human-subject data and you can't ship neuroimaging metadata off to a hosted API. That privacy constraint is why the zero-shot, no-fine-tuning result matters practically: a small lab can stand this up on the hardware it already has. And one counterintuitive detail — the mixture-of-experts models underperformed the smaller dense ones despite having more parameters, which is a useful reminder that raw model size isn't the axis that matters here. The ontology is.

So why feature an explicitly non-agentic paper on a show about agentic AI? Because the most useful thing an agentic-AI practitioner can hear right now is where agents aren't the answer. The framing everyone's converged on is that you compensate for a messy, under-specified data layer with cleverness in the orchestration layer — more agents, more retrieval, more self-correction. This is a clean, measured counterexample: invest that same effort one layer down, in an ontology written to be read, and a single language-model call does the job with no scaffolding at all. For the knowledge-graph and data-integration crowd — which is to say, a lot of us — that's not a limitation, that's the point. The ontology was always the product. This is a paper that finally measures how much.

That's NLKGQ.
