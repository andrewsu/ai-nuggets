# VFB-MCP — three generic tools plus ontology-driven query introspection over a Drosophila connectomics knowledge graph

Paper link: https://www.biorxiv.org/content/10.64898/2026.06.16.732577v1

## Script

Today's pick is VFB-MCP, a new bioRxiv preprint out of the Virtual Fly Brain team that exposes an expert-curated, ontology-backed Drosophila neuroscience knowledge graph to language models through the Model Context Protocol — and then, more importantly, actually benchmarks how much that plumbing changes what the model can answer.

If you were listening ten days ago I flagged TogoMCP, which was doing the same broad thing across the DBCLS RDF portal — twenty-three life-science databases behind a SPARQL layer. VFB-MCP is one database, not twenty-three, but the architectural contribution is different, and I think more interesting, so it's worth sitting with the design choice they've made.

Virtual Fly Brain is a Neo4j graph store, with the Drosophila Anatomy Ontology as its semantic backbone. Nodes are neurons, brain regions, cell types, genes, transgenes, publications. Edges are synapsed-to, part-of, subclass-of, expresses, has-image. It integrates the recent adult and larval connectomes — hundreds of thousands of neurons, tens of millions of synapses — together with single-cell transcriptomic atlases and transgenic driver collections. All the pieces you'd want, and traditionally all locked behind either a web browser or a Python API that assumes you already know how the graph is shaped.

Here is the design choice that makes this paper worth reading. The MCP server exposes only three tools. Not thirty. Not eighty. Three: search-terms, which does full-text search over the ontology; get-term-info, which fetches the canonical record for a given entity ID; and run-query, which executes a named query type against an entity. That is it. A tool surface that small usually cannot cover a knowledge graph's breadth, and here is the trick — get-term-info doesn't only return metadata, it returns a Queries array, the list of query types that are actually valid for that specific entity. Ask about a neuron, and get-term-info tells you the valid queries include upstream-class-connectivity, downstream-class-connectivity, parts-of, subclasses-of, expression-overlaps-here. Ask about a single-cell RNA-seq cluster, and get-term-info instead tells you cluster-expression is available. The ontology drives which tool calls make sense at runtime, per entity, per turn. There is no bespoke "give me dopamine-receptor expression for Kenyon cells" tool. There does not need to be one. Three generic primitives, one ontology-driven introspection step, and the semantics are carried by the graph, not the tool list.

That is a real design pattern, and it is the counterweight to the temptation of building an MCP surface that mirrors your database schema one tool to one query. You do not need every query wrapped as a distinct tool. You need entities to advertise, at fetch time, what can be asked of them.

The benchmark is where the intuition gets tested. Thirty tasks across four tiers — simple recall, integrative surveys, multi-hop reasoning, and quantitative graph traversal — scored by a Virtual Fly Brain curator against the database's current ground truth. Three conditions on Claude Opus four-point-seven: bare, MCP-augmented, and web-augmented, where web here means Claude Code's built-in WebSearch and WebFetch tools. The MCP arm gets precisely correct on twenty-five out of thirty tasks. Web gets fourteen. Bare gets two. On the hardest tier — connectivity reasoning, cross-modal queries, quantified synapse counts — MCP gets eight of nine right, web one of nine, bare zero.

The mechanistic story the authors offer is entity binding, and I think it lands. Traditional RAG, web-search RAG, and training-data recall all suffer from the same structural problem — they infer, from text, which entity a quantified claim is about. And language models get that wrong, confidently. A published paper describes a receptor-expression number for one Kenyon-cell subtype, and the model attaches the number to the wrong subtype. In an MCP call keyed on an entity identifier, that mistake is not available. run-query of entity A returns data about entity A. Misbinding is impossible by construction. The paper's failure-mode taxonomy is disciplined about this: with MCP, the scaffold is right and the errors are additions on top; with web, the scaffold itself is misassembled from the wrong sources. Frontier models do not fix this — Opus four-point-seven still misbinds on the web arm — because the failure is architectural, not a capability gap.

Two caveats worth carrying. First, cost. The MCP arm is roughly twelve times more expensive per task than bare, largely because a broad connectivity query can return over a megabyte of JSON and the harness spills it to disk and then reads it back through several turns of file bookkeeping. That is not a per-query cost you want to eat casually, and the authors flag it explicitly. Second, partial fabrication is not eliminated — the MCP-grounded scaffold stands up, but the model still occasionally fills in citations and peripheral details it invented from training data, and that failure mode is architecturally invisible to the tool layer. The MCP guarantees the entity binding, not the surrounding prose.

What I want you to hear on the way out is the framing the authors offer for where this goes. VFB-MCP is one node in an emerging biological MCP ecosystem — OmniPath for intercellular signalling, STRING for protein networks, UniProt for function, PubMed for literature — with ontology-backed knowledge graphs playing the role of the trustworthy scaffold and other MCP sources building on top. Same shape as what TogoMCP was gesturing at from the SPARQL side, but with a different answer to the "how does the model know what to ask" question — schema-file-in-context for TogoMCP, per-entity ontology-driven introspection for VFB-MCP. Both work. And the fact that both work, on cleanly measured benchmarks, is the substantive news of the last two weeks in this space.

That's VFB-MCP.
