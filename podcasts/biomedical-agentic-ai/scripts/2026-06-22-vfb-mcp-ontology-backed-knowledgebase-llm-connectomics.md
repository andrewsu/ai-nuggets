# VFB-MCP — an ontology-backed knowledgebase exposed over MCP makes an LLM precise on Drosophila connectomics

Paper link: https://www.biorxiv.org/content/10.64898/2026.06.16.732577v1

## Script

Today's pick is a bioRxiv preprint that landed yesterday from the Virtual Fly Brain consortium, led out of Edinburgh, and it sits exactly in the seam this show keeps circling. The system is called VFB-MCP. The headline is that they took Virtual Fly Brain, an expert-curated and ontology-backed knowledgebase of Drosophila neuroanatomy and connectomics, and exposed it to a large language model through the Model Context Protocol. The benchmark numbers are striking — but I want to walk through why those numbers are what they are, because the architecture is the read, not the score. One caveat up front: bioRxiv hasn't rendered the body HTML for this preprint yet, so I'm going off the abstract — the design and the headline benchmark numbers are unambiguous, but I haven't seen the full methods.

Setup. There is a well-known mismatch between how biological knowledge is curated and how a working scientist needs to query it. Virtual Fly Brain has spent years building the ontologies and the cross-linkage that let you ask, in a structured query language, things like "show me every neuron downstream of this olfactory projection neuron in the hemibrain connectome, and tell me whether any of them are also expressed in this Janelia driver line." That structured query is precise — and it's gated by domain expertise. Most new entrants to the field don't know the query languages, don't know the ontology, and don't know which of half a dozen integrated datasets actually contains the answer. The pre-LLM solution is a web interface. The post-LLM solution most groups are trying is to point ChatGPT at the database and hope. VFB-MCP is a third option that I think is the right shape.

What they did. They wrote a Model Context Protocol server that wraps Virtual Fly Brain's underlying knowledge — the ontology, the curated annotations, the integrated connectomes — into a set of tools an LLM can call at inference time. The model doesn't see a SQL-equivalent query language. It sees a menu of typed operations: look up a neuron, traverse a connectome edge, fetch ontology relationships. When a user asks a natural-language question, the LLM decomposes it into a sequence of these typed calls, the MCP server hits the ontology-backed knowledgebase, and the LLM composes the structured returns into an answer. The key word in that pipeline is "ontology-backed" — every entity the model can talk about has a curated identity. The model isn't generating ontology terms from its head; it's binding to ones the curators put there.

The benchmark. Thirty neuroscience tasks, run in three configurations: a bare frontier LLM, a web-search-assisted LLM — that is, retrieval over the open web — and the VFB-MCP-equipped LLM. The bare model gets two out of thirty right. The web-search model gets fourteen out of thirty. The MCP-equipped model gets twenty-five out of thirty. The pairwise differences are all significant at p less than point oh one with Holm correction. And the most interesting slice — the place where the gap widens dramatically — is on tasks that require data quantification. There, MCP hits eighty-nine percent and the web-augmented model hits eleven. Eleven. Connectome counts and ontology traversals are not the kind of thing the open web can paraphrase its way through. Either you have a precise structured handle on the data, or you don't.

The Su Lab read. Three things, and they're the reason I picked this over a multi-agent fluorescence microscopy framework that posted to bioRxiv on the same day.

First, this is a clean demonstration of the right division of labor for biomedical knowledge graphs in the LLM era. The expert curators do what they're irreplaceable at — building and maintaining the ontology, the cross-references, the data integration. The LLM does what it's irreplaceable at — natural language understanding, intent parsing, composing an answer in the user's words. The MCP server is the contract between them. Neither side gets asked to do the other side's job. Compare this to the dominant alternative pattern in this space, which is fine-tuning a domain LLM on a knowledge graph dump and then asking the LLM to be both the natural language interface and the source of truth. That collapses the curation and the inference into the same set of weights, and you lose updatability, you lose verifiability, and — as the eleven-percent quantification number on web-augmented LLMs shows — you lose precision on exactly the queries the database existed to answer.

Second, and this is the operational reason this matters now: MCP is the abstraction layer that lets every curated biomedical resource expose itself this way without picking a winning model vendor. The same VFB-MCP server works with whatever LLM the user has access to next year. The investment is in the server, not the model. For anyone running a biomedical knowledge graph — Wikidata's biomedical content, the BioPortal ontologies, ChEMBL, the dozens of model-organism databases — the lesson is that the right next engineering project is an MCP wrapper, and the right next quality bar is whether your wrapper makes an off-the-shelf LLM hit ninety percent on quantification questions the open web bombs.

Third, and this is the part that should bother anyone who has built one of these systems before: the benchmark is thirty hand-curated neuroscience tasks. That's small. It's small enough that the authors could have designed around what their MCP tools handle well. The honest version of this paper has a community-extended benchmark that other groups contribute tasks to, ideally including failure modes the authors didn't anticipate. The architectural claim is strong; the validation set is one team's reading of what hard means. I'd take this as a strong existence proof of the pattern, not yet as a settled measurement of the ceiling. That's it for today — talk to you tomorrow.
